# Alchemist Hunter 개발 기획 v1

## 0. 목적
- IdleGuildMaster의 강점(장기 성장 루프, 반복 파밍 동기, 클래스/특성/펫 조합)을 참고하되, 그대로 복제하지 않고 `Alchemist Hunter`만의 코어 루프를 구현한다.
- 목표는 “출시 가능한 최소 게임(MVP)”을 빠르게 만들고, 이후 라이브 업데이트로 확장하는 것이다.

## 1. 제품 방향

### 1.1 코어 루프
- 영웅 편성 -> 던전 자동 전투 -> 재료/장비 획득 -> 성장(레벨/장비/특성/펫) -> 상위 던전 해금.

### 1.2 핵심 차별점(초안)
- 클래스 트리보다 `연금 시너지`를 강조:
  - 장비 + 촉매(Alchemy Catalyst) 조합으로 임시 버프/상태 이상 빌드 생성.
- 어둠 수치와 유사한 환경 변수:
  - 던전마다 `오염도(Corruption)`를 도입해 전투 규칙 변화.

### 1.3 MVP 기준
- 플레이어가 “자동 전투 + 성장 + 해금”을 30분 이상 반복할 수 있어야 한다.
- 최소 콘텐츠:
  - 던전 5개
  - 영웅 클래스 4종
  - 장비 40개
  - 펫 6종
  - 특성 12개

## 2. 기능 범위

### 2.1 전투/던전
- 턴제 자동 전투 엔진(스킬 쿨다운, 상태 이상, 위협도 타게팅).
- 던전 진입 조건(클리어 횟수/보스 처치/특수 아이템).
- 던전 환경 변수(오염도, 버프/디버프 룰).

### 2.2 성장
- 영웅 레벨/승급(티어) 시스템.
- 장비 드롭 + 제작식 + 상위 장비 승급.
- 특성 2슬롯(1차 특성, 2차 특성)과 조합 효과.
- 펫 레벨업 + 펫 특성.

### 2.3 경제/보상
- 재화 3종: 골드, 다이아, 연금석.
- 드롭 테이블(확률/중복 드롭/희귀도).
- 오프라인 보상(누적 시간 기반, 상한 존재).

### 2.4 UI
- 하단 4탭 유지: Characters / Weapons / Dungeons / Pets.
- 리스트 기반 -> 상세 -> 강화/편성 흐름.

### 2.5 저장/동기화
- 로컬 저장 우선(Hive 또는 Isar).
- 계정 연동은 v2 범위.

## 3. 기술 설계

### 3.1 아키텍처
- `Flutter + Riverpod` 기반.
- 권장 폴더 구조:
  - `lib/core` 공통(에러, 유틸, 상수)
  - `lib/features/{feature}/{data,domain,presentation}`
  - `lib/app` 라우팅, 테마, 부트스트랩

### 3.2 현재 코드 상태 기준 선행 정리
- `lib/main.dart`와 `lib/app/*` 엔트리 구조가 분산되어 있으므로 단일 엔트리로 통합 필요.
- `lib/features/dungeon/model.dart`, `lib/app/app.dart`, `lib/app/bootstrap.dart`가 비어있어 초기 골격부터 재작성 필요.

### 3.3 상태관리 원칙
- 화면 상태: `StateNotifier/Notifier`.
- 게임 핵심 상태(파티, 인벤토리, 던전 진행): Feature별 Provider 분리.
- 전투 로직은 UI와 완전 분리하여 테스트 가능하게 설계.

## 4. 데이터 모델(초안)

### 4.1 Hero
- `id`, `name`, `classId`, `tier`, `level`, `stats`, `traits`, `equippedItems`, `petId`.

### 4.2 Dungeon
- `id`, `name`, `unlockRule`, `environment`, `waveSet`, `dropTable`.

### 4.3 Item
- `id`, `type`, `rarity`, `stats`, `sellPrice`, `recipe`, `source`.

### 4.4 Pet
- `id`, `name`, `traits`, `level`, `exp`.

### 4.5 Combat Snapshot
- `turn`, `units`, `statuses`, `logEvents`.

## 5. 마일스톤 (6주)

### M1 (1주차): 기초 뼈대
- 앱 엔트리 통합, 라우팅/테마 정리.
- 더미 데이터 기반 탭 동작 안정화.
- 저장소 추상화 인터페이스 정의.

### M2 (2주차): 전투 최소 구현
- 자동 턴 전투 엔진 v0.
- 캐릭터/몬스터 기본 스탯 및 일반 공격.
- 던전 2개 플레이 가능 상태.

### M3 (3~4주차): 성장 루프 완성
- 드롭/인벤토리/장비 착용.
- 경험치/레벨업/승급.
- 오프라인 보상 로직.

### M4 (5주차): 확장 콘텐츠
- 특성/펫 시스템 추가.
- 던전 해금 조건/환경 변수 적용.

### M5 (6주차): 안정화
- 밸런스 1차 튜닝.
- 저장 데이터 마이그레이션 대응.
- QA 체크리스트 및 내부 테스트 빌드.

## 6. 리스크 및 대응
- 밸런스 붕괴:
  - 대응: 시뮬레이터성 내부 툴(전투 1,000회 자동 돌려 평균 DPS/생존율 산출).
- 상태 복잡도 증가:
  - 대응: 전투 로그/리플레이 데이터 구조를 초기에 고정.
- 데이터 수정 난이도:
  - 대응: 하드코딩 최소화, JSON 기반 밸런스 데이터 로딩.

## 7. 즉시 착수 작업 (다음 커밋 단위)
- `app 엔트리 통합`:
  - `main.dart` -> `app/bootstrap.dart` 호출 구조로 일원화.
- `기본 도메인 모델 생성`:
  - Hero, Dungeon, Item, Pet 엔티티 파일 추가.
- `던전 탭 실제 데이터 연결`:
  - 현재 더미 `Dungeon Name` 리스트를 모델 기반 리스트로 교체.
- `기본 전투 엔진 인터페이스`:
  - `CombatEngine.run(input)` 시그니처와 결과 모델 정의.
